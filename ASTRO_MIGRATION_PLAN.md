# 484.sk → Astro Migration Plan

**Goal:** Solve the #1 SEO blocker (no per-listing URLs, no server-rendered HTML) without rewriting the whole stack. Keep your CSS, your modules, your Supabase backend, your admin panel. Only the public site (`index.html`) gets rebuilt.

**Estimated effort:** 1 weekend for MVP, 1 more weekend for polish.

**Why Astro (not Next.js):**
- Static-first: pulls listings from Supabase at *build time*, emits one HTML file per listing. No Node server needed in production. Vercel/Netlify static hosting still works.
- Zero JS by default. You opt into hydration (`client:load`, `client:visible`) only where you need it (search, gallery, language switcher).
- Page-speed weapon: Lighthouse routinely scores 95+ out of the box.
- Migration path: you can copy your existing CSS and JS modules into Astro components almost verbatim. No React rewrite.
- Component islands: your existing vanilla JS modules (`SubmitListing`, `Gallery`, `LangEngine`) drop in as `<script>` tags inside Astro components.

## Target URL structure

| Current (SPA) | New (Astro) |
|---|---|
| `484.sk/` (everything) | `484.sk/` — listings index |
| (no URL — JS-only `goDetail()`) | `484.sk/inzerat/{id}-{slug}/` — per-listing detail page |
| (no URL) | `484.sk/typ/dom/` — category pages by property type |
| (no URL) | `484.sk/kraj/kosicky/` — category pages by region |
| `?lang=en` (URL param) | `484.sk/en/`, `484.sk/de/`, `484.sk/uk/` — proper i18n routes |
| `?q=foo` (search) | Same — client-side filter on listings index |

Every URL above is a real, server-delivered, fully-rendered HTML page that Googlebot indexes on first crawl. Per-listing JSON-LD, Open Graph, and breadcrumbs are baked into the HTML at build time.

## Build-time data flow

```
Supabase (inzeraty table)
        ↓
src/pages/inzerat/[slug].astro  (getStaticPaths)
        ↓
fetch all active listings → emit one HTML file per row
        ↓
dist/inzerat/{id}-{slug}/index.html  (deployed to Vercel)
        ↓
Googlebot crawls → indexes price/photos/description directly from HTML
```

**Critical:** rebuild on every admin approve/reject. Wire a Supabase Edge Function or DB trigger → POST to Vercel/Netlify deploy hook → rebuild + redeploy in ~30 seconds.

## Phased migration (don't big-bang it)

### Phase 1 — Astro skeleton + listings index (½ day)
1. `npm create astro@latest mokrance-astro` (use minimal template)
2. Add `@astrojs/sitemap`, `@astrojs/check`
3. Port `index.html` `<head>` into `BaseLayout.astro`
4. Build `pages/index.astro` that fetches listings from Supabase at build time and renders `ListingCard.astro` per row
5. Verify it builds, deploy to Vercel preview

### Phase 2 — Per-listing pages (½ day)
1. Build `pages/inzerat/[slug].astro` with `getStaticPaths` returning `{slug: "{id}-{seo-slug}"}` for every active listing
2. Inject per-listing JSON-LD (`Product`, `BreadcrumbList`) at build time — no more JS injection
3. Set per-listing `<title>`, `og:*`, canonical, hreflang at build time
4. Verify URLs like `/inzerat/123-rodinny-dom-mokrance/` return full HTML

### Phase 3 — Hydrated SPA bits (1 day)
1. Port `SearchInput`, `FilterRail` as `client:load` components — same JS logic as today, just scoped
2. Port `Gallery` as `client:visible` — only loads JS when user scrolls to it
3. Port `SubmitModal` as `client:idle` — only loads JS when browser is idle
4. Port `LangEngine` as a shared client module

### Phase 4 — Multilingual routes (½ day)
1. Use Astro's i18n routing → `/en/`, `/de/`, `/uk/` get pre-rendered with translated copy
2. Pull translations from Supabase columns (`nazov_en`, `popis_de`, etc.) at build time
3. Wire hreflang tags per page

### Phase 5 — Build automation (¼ day)
1. Vercel deploy hook URL → store as Supabase secret
2. Edge Function on `submitted_listings.status = 'approved'` → POST to deploy hook
3. Manual "rebuild" button in admin panel as fallback
4. CI check: weekly cron rebuild even with no listing changes (catches drift)

### Phase 6 — Cutover (½ day)
1. Deploy Astro build to staging URL
2. Verify all pages render correctly with full HTML (curl + grep)
3. DNS cutover: point `484.sk` to Astro deployment
4. Old `index.html` becomes a fallback on a subdomain (`old.484.sk`) for rollback safety
5. Submit new sitemap to Google Search Console → request reindex

## Risk mitigation

- **Rollback plan:** keep current SPA deployed at `old.484.sk`. If Astro version breaks anything in production, flip DNS back in 5 minutes.
- **Build failures:** wrap Supabase fetch in `getStaticPaths` with try/catch + fallback to last successful build. Never let one bad row break the entire deploy.
- **SEO transition:** 301 redirect old SPA URL params (`?listing=123`) to new clean URLs (`/inzerat/123-slug/`). Preserves any existing link equity.
- **Photo URLs:** Storage bucket URLs don't change. No image migration needed.

## What stays exactly as-is

- `admin.html` — unchanged, deployed as a static asset under `/admin.html`
- `config.js` — unchanged, copied into `public/`
- Supabase schema — zero changes
- Photo Storage buckets — zero changes
- DeepL Edge Function — unchanged
- Telegram notifications — unchanged

The migration is **pure rendering layer**. Backend stays put.

---

## Decision points before starting

1. **Hosting:** Vercel (recommended — best Astro integration) vs Netlify vs Cloudflare Pages. All work. Vercel has the smoothest deploy hooks.
2. **Build trigger:** webhook on every admin approve, or scheduled (e.g. every 15 min)? Recommend webhook for freshness.
3. **URL slug format:** `{id}-{nazov-slug}` (e.g. `123-rodinny-dom-mokrance`) vs pure ID (`123`). Recommend slug format for keyword-rich URLs.
